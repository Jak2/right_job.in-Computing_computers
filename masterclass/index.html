The 20-Minute Geek Core

Not memorized lines — a way of thinking that leaks into every answer and makes the interviewer think this person actually builds these things.

§1The one belief that runs through every answer

An LLM is a brilliant, unreliable component. My job is not to trust it — my job is to build a system around it that stays correct even when the model is wrong.

Say this in your own words when they ask anything open-ended. Every design decision you made — the 0.65 threshold, the AND-gate, the regex fallback, the blind judge, the verifier-retry loop — is the same idea pointed at a different failure. That single belief is your senior signal. A junior asks "what can the model do?" A senior asks "what happens when it fails, and who gets hurt?"

Framed this way, you're not an engineer who uses AI. You're an engineer who makes AI safe to put in front of a paying customer. That's the whole job.

§2Six ideas that make you sound like a builder, not a reader

1 — Retrieval is a filtering problem before it's a similarity problem
You filter at the database level before computing similarity. A Nokia error must never pull an Ericsson fix — wrong CLI syntax could make a live outage worse. Qdrant's must filter runs before ranking; FAISS can't do that — you'd search the whole corpus and discard wrong-vendor matches afterward. Metadata pre-filter beats post-filter. One sentence, sounds like you've run RAG at scale.
2 — Faithfulness ≠ accuracy
Accuracy asks "is the answer true?" Faithfulness asks "did the model stay inside the documents I retrieved?" High faithfulness + wrong answer = retriever's fault (bad docs). Low faithfulness = model ignored the retriever and used its own memory. Different failure, different fix. You built a scorer that separates these — almost no candidate can articulate this cleanly.
3 — Quality gates use AND logic, never a weighted average
A response that's 95% faithful but hallucinates one claim must fail. An average hides exactly the number that hurts a user. Your eval framework fails a test case if any single evaluator drops below threshold. "It's a quality gate, not a quality summary." Apply this mindset to guardrails, CI checks, agent verifiers — everything.
4 — The right answer is usually "the cheapest model that reliably does the task"
Model selection is a design decision, not a default. Haiku for narrow, well-defined extraction; Sonnet for ambiguous reasoning. Your 5G router: high-similarity + non-critical → Haiku; ambiguous or critical → Sonnet. ~60% cost cut, zero accuracy loss on hard cases. "I'd just use GPT-4 for everything" is a junior tell. Routing is the senior move.
5 — Every agent needs a refusal path
The hardest part of shipping agents isn't making them act — it's making them not act when they shouldn't. Below 0.65 similarity, your pipeline says "unknown — human review" instead of inventing advice. "An agent without an 'I don't know' path is a liability, not automation." This is your answer to "what's hard about production agents?"
6 — Memory is a data-access-pattern problem, not a magic feature
Three kinds: working memory (hot, changes fast → Redis + TTL), episodic history (cold, survives restarts → SQLite/Postgres), semantic memory (knowledge that grows → vector DB). NFTRACE's Redis+SQLite split is working memory + episodic history; the unknown_errors Qdrant collection is semantic memory growing from the agent's own failures. Right store per access pattern.

§3The agent loop — draw it in your head, describe it in one breath

Every agent, including their "UWM", is a loop: Perceive → Retrieve → Reason → Act → Observe → (escalate if unsure) → repeat.

Your 5G pipeline maps onto this exactly: Pyshark perceives, Qdrant retrieves, Claude reasons, the system acts, confidence gates escalation. That IS an agent — you don't need their framework's name to have built the pattern.

The academic name for the tight inner loop is ReAct: Reason → Act → Observe, looping, with the model alternating between thinking and tool calls, feeding each observation back into context. LangChain agents, tool-use loops, most agent frameworks are ReAct underneath. Function/tool calling is the mechanism: the model returns a structured {name, args} instead of prose, your code runs it, you feed the result back. MCP standardizes how those tools are exposed to the model. That's the whole stack in four sentences.

§1AI Agent Architecture & the Agent Loop

Lead with: an agent is a loop with a decision at runtime. A workflow is a fixed sequence you designed; an agent decides its own next step from current state. Canonical loop: perceive → retrieve context → reason → act (call a tool) → observe → decide whether to loop, finish, or escalate. The tight academic version is ReAct — reason/act/observe, model alternates thinking and tool calls, each observation feeds back into context.

Senior vs junior: juniors describe the happy path. You talk about control — stopping infinite loops, capping steps, detecting when the agent is stuck, and when it refuses to act. "Every agent I've shipped has an explicit 'I don't know' path. An agent that always acts is a liability."

Failure modes you raise unprompted

  • Runaway loops — cap max iterations; detect repeated identical tool calls (the model is stuck).
  • Context bloat — each observation appends; window fills, cost climbs, quality drops. Manage what enters the prompt (§7).
  • Confidently wrong actions — validate tool inputs/outputs; keep a human gate on anything irreversible.
Q — Agent vs workflow?
"A workflow is a fixed sequence — I decide the steps at design time. An agent decides its next step at runtime based on state. My 5G pipeline is a hybrid, which I think is the honest production answer: agentic at the decision points — match-confidence chooses auto-fix, deeper reasoning, or human escalation — deterministic everywhere else. Pure autonomy everywhere is usually a liability."
Q — How do you keep an agent from looping forever?
"Three guards: a hard iteration cap, a loop-detector that catches the same tool call twice, and a confidence gate that escalates instead of grinding. The agent should fail loudly and safely, not silently and confidently."
Your proof: Brane agent loop, tool registry context, verifier-retry correction loop · 5G pipeline full perceive→retrieve→reason→act→escalate loop in production.

§2Multi-Agent Systems, A2A & Orchestration

Multi-agent = decomposing a hard job into specialized agents that coordinate. Two shapes: coordinator/orchestrator (hub delegates to sub-agents, aggregates results) and peer-to-peer A2A (agents talk directly, negotiate, hand off). The hard problems mirror distributed systems: shared state, failure isolation, decision authority.

Honest framing — don't bluff: "I've built coordinator-pattern systems in production. My eval framework runs four independent evaluator agents whose verdicts get aggregated by AND-logic. My 5G pipeline is a staged pipeline with human-escalation authority. I haven't shipped peer-to-peer A2A protocols yet, but the genuinely hard parts — shared state, failure isolation, verdict aggregation — are ones I already solve. I'd want to hear how UWM structures A2A messaging."
Q — How would you aggregate outputs from agents that disagree?
"Depends on risk. For a quality gate, AND-logic — any failure blocks the result, false alarm beats missed defect. For a best-answer scenario, weighted voting or a judge agent. The anti-pattern is averaging — it hides the one dissenting signal that mattered."
Your proof: eval framework = 4-agent coordinator, AND aggregation · 5G pipeline = staged pipeline with escalation authority.

§3RAG Pipelines & Vector Databases

RAG in one breath: chunk knowledge → embed each chunk → store in a vector DB → at query time embed the query, retrieve top-k nearest chunks, stuff into the prompt as grounding, generate from that context. The point: fresh, private, citable knowledge the model wasn't trained on.

Nokia patterns never enter the Ericsson similarity computation — the must filter runs before ranking, at the DB level. Below threshold, the agent refuses rather than hallucinates, and its own failure becomes new training data.

The two insights that make you sound expert

1 — Metadata pre-filter beats post-filter
Retrieval is a filtering problem before it's a similarity problem. Qdrant's must filter runs at the DB level, before similarity is computed. FAISS has no metadata filtering — you'd search the whole corpus then discard wrong-vendor matches, slower and it pollutes the ranking.
2 — Faithfulness ≠ accuracy
See geek core §2. This is what you say when RAG quality comes up.

Design decisions you can defend

  • Cosine, not Euclidean — measures angle/direction (concept), not magnitude (verbosity). "T3560 timeout" and the verbose version point the same direction.
  • Threshold 0.65 — below it, a low-confidence answer is hallucinated advice, worse than nothing. Refuse, log, learn.
  • return_source_documents=True — trust. An ops engineer about to restart a live service needs "matched pattern AUTH_TIMEOUT_T3560, 0.94 similarity," not a black box.

Chunking — have an opinion

Too-large chunks dilute the embedding (one vector, many ideas, poor recall); too-small lose context. Chunk on semantic boundaries, overlap slightly so a fact split across a boundary still retrieves. "Bad chunking is the silent killer of RAG — people blame the model when retrieval never surfaced the right passage."

Vector DB landscape — know the tradeoffs

DBTradeoff
QdrantDB-level metadata pre-filtering, production-grade — what you used.
FAISSA library, blazing fast, in-memory — no metadata filtering, no persistence. Fine for a static index, wrong for multi-tenant.
pgvectorVector search inside Postgres, next to relational data. "For a team already on RDS Postgres — this JD — pgvector removes an entire piece of infrastructure. I'd put it on the table here."
Pinecone / WeaviateManaged, scale-out — trade cost/lock-in for not running it yourself.
Q — How would you debug a RAG system giving wrong answers?
"I don't start at the model — I start at retrieval, because that's where most RAG bugs live. Log the retrieved chunks: did the right passage come back? If not, it's retrieval — bad chunking, wrong embedding model, missing metadata filter. If it did come back and the answer's still wrong, it's generation — faithfulness score tells me whether the model ignored context. Juniors blame the LLM first; nine times out of ten it's retrieval."
Your proof: 5G pipeline — full production RAG, Qdrant pre-filtering, cosine, threshold + refusal, source docs, LangChain RetrievalQA.

§4LLM Integration & Model Routing

You don't integrate "an LLM," you integrate a fleet and route between them. Cheap/fast for narrow well-defined tasks, expensive/deep for ambiguous or high-stakes reasoning. Provider-agnostic where possible, so one vendor's outage or price hike doesn't sink you.

Routing philosophy: "Model selection is a design decision, not a default." 5G router: top_score > 0.92 AND score_gap > 0.15 AND severity != critical → Haiku; else → Sonnet. ~60% cost cut, no accuracy loss on hard cases. Keyword generator: Haiku only — narrow extraction, 3–5× cheaper, ~500ms.

Temperature is a correctness knob, not a vibe: 0.1 (or 0) for operational tooling — same input, same output. High temperature is actively harmful when an engineer running the same pcap twice must get the same fix. Raise it deliberately when you want variance — your consistency checker re-runs at 0.7 to measure output stability.

The reliability layer

  • Structured output validation — model returns JSON, validate with Pydantic before trusting it. Malformed output is a when, not an if.
  • Fallback chains — LLM → regex → sane defaults. "The LLM path is an enhancement, not a dependency." How the keyword generator claimed 100% uptime.
  • Response parser hardening — Brane: hardened the parser for malformed output, built a verifier-with-retry loop feeding the error back for correction.
  • Rate limiting — cap in-flight calls with asyncio.Semaphore (bias auditor caps at 10; without it, 500 calls die instantly on provider limits).
Q — How do you deal with hallucinations?
"You can't make the model stop hallucinating — you build a system that catches it. Grounding via RAG, faithfulness scoring to detect straying from context, structured-output validation, confidence thresholds with a refusal path, human-in-the-loop on anything irreversible. I treat the model as an unreliable dependency and wrap it in the same discipline as any flaky external service."
Your proof: 5G model routing · Brane multi-provider gRPC gateway (instruct/thinking/vision routing, stateless horizontal scaling) · keyword-generator fallback chain.

§5Prompt Engineering

A production prompt has structure: system persona, grounded context (retrieved facts, injected input), tool registry (what it can call), the task, and an output contract (exact schema). You're assembling a prompt, not writing a sentence.

  • Grounding / context injection — retrieved facts in the prompt, answer from them not memory. The anti-hallucination workhorse.
  • Few-shot examples — 2–3 input→output pairs when the task shape is unusual.
  • Output contracts — exact JSON schema + Pydantic validation. Never parse prose you could've gotten structured.
  • Retry-with-feedback — failed validation feeds the specific error back: "your JSON was missing field X." Built at Brane.
  • Prompt caching — stable content (system prompt) first, variable content last — cheaper, faster on repeated calls.
Q — How do you make an LLM reliably return structured data?
"Ask for JSON with an explicit schema, temperature zero, validate with Pydantic instead of trusting it. On failure I don't just retry blindly — I feed the validation error back so the model corrects the specific mistake. And I always keep a deterministic fallback so a malformed response degrades gracefully instead of crashing."
Your proof: Brane prompt assembly + verifier retry loop · keyword generator temperature=0 JSON extraction with regex fallback.

§6LangChain / LangGraph

LangChain — used in production (5G pipeline's RetrievalQA). Composable blocks: chains, retrievers, memory, output parsers. Value = swappability — swap LLM or retriever in one line, return_source_documents=True gives citations free. Senior critique: "great for standing up RAG fast, but abstractions get leaky at scale — sometimes you drop to raw API calls for control over exactly what enters the context window."

LangGraph, owned conceptually: "LangChain's answer to stateful, cyclic agent orchestration. Model the agent as a graph — nodes are steps, edges are control flow, supports cycles, which real agent loops need. Two features that matter: explicit shared state between nodes, and checkpointing — pause the graph, get a human decision, resume. That checkpoint IS human-in-the-loop built in. My Celery pipelines follow the same DAG thinking."
Q — Why LangGraph over a plain LangChain agent?
"Fine for a simple ReAct loop, but once you need branching, cycles, persistent state, or a human approval gate mid-run, you want LangGraph's explicit graph — control the flow instead of hoping the agent's reasoning takes the right path."
Your proof: LangChain RetrievalQA in the 5G pipeline with source documents.

§7Memory, Context Handling & Agent Lifecycle

Best one-liner: "Agent memory is a data-access-pattern problem." Three kinds, three stores:

NFTRACE's Redis+SQLite split is working memory + episodic history. The 5G pipeline's unknown_errors collection is semantic memory — "the agent's failures literally become its training data."

Context-window management: the window is finite, every token costs money and dilutes attention. Prioritize: system persona + top-k retrieved chunks (not the whole corpus, top-5 above threshold) + recent turns; summarize or evict old history; stable content first for caching.

Agent lifecycle: created → running → waiting-on-human → completed/failed. Persist state per run (Brane's agent_run_scratch, agent_episodes), restart-safe (Redis persists where an in-memory dict wouldn't), TTL orphaned sessions so a crashed agent doesn't hold resources forever.

Q — How do you manage context when a conversation gets too long?
"Can't just keep appending — you'll blow the window and quality drops as it fills. I keep the system prompt and top-k retrieved facts, keep recent turns verbatim, summarize or drop older history. The skill is deciding what's load-bearing."
Your proof: NFTRACE Redis+SQLite temperature split · Brane agent_run_scratch/agent_episodes · 5G growing unknown_errors semantic memory.

§9Python (Advanced) — Async & Concurrency

Pick the concurrency model that matches the bottleneck. I/O-bound (LLM calls, DB, network) → asyncio — one thread juggling many waits. CPU-bound (parsing, math) → multiprocessing — true parallelism across cores, since the GIL serializes threads. Threads: I/O-bound work stuck with sync libraries.

  • asyncio + Semaphore — bias auditor: asyncio.gather with a semaphore capping in-flight calls at 10.
  • multiprocessing.Pool — NFTRACE parallel node ops; xmlrpc.client is synchronous, traces on N nodes must start simultaneously or you miss the first messages. asyncio would've meant an async XML-RPC dependency; overhead negligible for 5–10 nodes.
  • Celery + Redis — durable background tasks, survives a restart mid-job. Non-negotiable for a CI/CD quality gate.
Q — asyncio vs multiprocessing vs threading — when each?
"Comes down to what you're waiting on. I/O-bound — LLM APIs, DB, network — asyncio, mostly waiting, one event loop juggles thousands of waits cheaply. CPU-bound — heavy parsing, numerical — multiprocessing, GIL means threads won't run bytecode in parallel. Threading only for I/O-bound work trapped behind a sync library. Most AI backend work is I/O-bound, so I default to async."
Your proof: async semaphore (bias auditor) · multiprocessing (NFTRACE) · Celery (eval framework).

§10PostgreSQL — Indexing, Query Optimization, Schema for AI Workloads

Indexes: a sorted lookup structure (B-tree by default) turning a full-table scan — O(n) — into a logarithmic seek. Always mention the tradeoff unprompted: indexes speed reads but slow writes (every insert/update maintains the index) and cost storage. Index columns you filter/join on, not everything.

  • EXPLAIN ANALYZE is your first move. "I don't guess why a query is slow — I read the plan. Seq Scan where it should be an Index Scan, where time actually goes, whether row estimates are off — usually stale statistics, fixed with ANALYZE."
  • Composite index column order matters — (vendor, layer) serves queries filtering on vendor or vendor+layer, not layer alone. Order by selectivity and actual query patterns.
  • Beyond B-tree — GIN for JSONB/full-text, partial indexes for "only the rows I query," and for AI workloads, pgvector's HNSW/IVFFlat for approximate nearest-neighbor.
  • When an index doesn't help — low-cardinality columns, small tables, queries returning most of the table.

Schema for AI workloads (Brane): agent identity tables, multi-agent junction tables for many-to-many agent↔LO relationships, attribute-level RBAC permission rows, execution sibling tables (lo_input_execution_agent, agent_run_scratch, agent_episodes). Attribute-level RBAC = permissions down to individual attributes, not coarse roles — cached because it sat on a hot path.

Q — A query got slow in production — walk me through fixing it.
"First, EXPLAIN ANALYZE — the actual plan, not a theory. Nine times out of ten it's a Seq Scan on a column that should be indexed, or a join missing an index on the key. If estimates look wrong, ANALYZE refreshes statistics. For a filtered query, a composite or partial index matching the exact predicate. Then re-run EXPLAIN ANALYZE to confirm the plan changed and time actually dropped — I verify the fix, I don't assume it."
Your proof: Brane PSQL schema (junction tables, attribute-level RBAC with caching, execution tables) · Redis caching of hot permission checks.

§11REST API Design, FastAPI & Microservices

REST = resources (nouns) + verbs (GET/POST/PUT/DELETE), stateless, meaningful status codes. FastAPI's edge: async-native, Pydantic validation at the boundary, auto-generated OpenAPI docs.

Long-running work pattern: an agent task takes 5–30s — don't block the request. Return a job ID immediately (201/202), process in the background (Celery), client polls or gets pushed results via WebSocket/SSE. Built exactly this in the eval framework.

Microservices judgment: "They buy independent scaling and deployment at the cost of network latency and distributed complexity. I split when parts have genuinely different scaling profiles — a stateless model gateway separate from a stateful orchestrator — not because it's fashionable. NFTRACE stayed a well-structured monolith because its deployment profile didn't justify the overhead."

Your proof: FastAPI across every project · Brane stateless gRPC gateway (horizontal scaling) · NFTRACE REST lifecycle API · job-ID + poll pattern (eval framework).

§12Auth — OAuth2, JWT & API Security

Authentication = who are you; authorization = what can you do. JWT — signed, self-contained, server verifies without a DB lookup (stateless, scales horizontally, harder to revoke early — keep short-lived + refresh tokens). OAuth2 — the delegation framework (authorization-code for users, client-credentials for service-to-service).

Security practices: validate every input at the boundary (Pydantic), least-privilege (IAM roles at Abjayon), secrets never in code, HTTPS everywhere, rate limiting — and the agent-specific one, attribute-level RBAC so an agent only touches attributes its role permits. Maps directly to the JD's healthcare/HIPAA "who can touch what data."

Q — JWT vs session tokens?
"Session tokens are stateful — instant revocation, doesn't scale as cleanly across servers. JWTs are stateless and self-contained — great for horizontal scaling since any instance verifies the signature, but you trade away easy revocation, so keep them short-lived with refresh tokens. For a multi-service agent platform, JWT's statelessness usually wins."
Your proof: Abjayon IAM roles + secure resource access · Brane attribute-level RBAC.

Before MCP, every app wired tool-calling its own way; MCP standardizes the interface so any MCP-compatible model can discover and call any MCP server's tools. "It's the natural next step from the LangChain tool integrations I've done — same idea, standardized so it's portable across models and apps." You use Claude/MCP tooling day-to-day (Claude Code, on your resume).

Q — What problem does MCP solve?
"Fragmentation. Everyone was building bespoke tool-calling glue between their app and the model. #1 ★5G Failure Detection & Remediation Pipeline

PysharkQdrantLangChain RetrievalQAClaudeRedis

Why it's your lead: hits five JD bullets in one story — autonomy, RAG, HITL, escalation, self-improvement. It IS an agent loop. (Full pipeline diagram in Tech Deep-Dives §3.)

"Picture a 5G core test lab where every signaling failure — an authentication timeout, a dropped handover — meant an engineer hand-scrolling gigabytes of packet captures for hours. And the frustrating part: the same failures kept recurring, diagnosed from scratch every time.

So I built a pipeline that does the whole loop autonomously. Pyshark extracts the signaling error, Qdrant retrieves the closest known patterns, Claude — grounded on those patterns — synthesizes a contextual fix.

The key decision was how it recognizes errors. Keyword rules are brittle — the same fault shows up worded differently every time. So I matched semantically against a vector database, pre-filtered by vendor and layer at the database level, so a Nokia error never pulls an Ericsson fix — wrong CLI syntax, could make a live outage worse.

The part I'm proudest of: what happens when the agent doesn't know. Below 0.65 similarity, a confident answer would be hallucinated advice — dangerous on a live network. So it refuses, escalates to a human, and logs the error so ops engineers curate new patterns. The agent's failures become its training data.

Result: 70% of routine errors auto-resolved, 55% MTTR reduction, cited in three internal audits."

One analogy, max: semantic search = "a librarian who finds the right book even when you misremember the title."

Go-deep answers

  • Why Qdrant over FAISS? Metadata pre-filtering at the DB level — Ericsson never enters the similarity computation for a Nokia error.
  • Why cosine? Angle not magnitude — concept over verbosity.
  • Why temperature 0.1? Same pcap must give the same fix. Determinism is a feature for ops tooling.
  • Why not just return the closest document? The document is generic ("check T3560 timer"). Claude contextualizes to this instance, synthesizes across partial matches: "this happened during a traffic spike — raise T3560 from 1s to 3s in the AMF config and monitor."
  • Redis? Caches frequent lookups so recurring failures resolve near-instantly.
  • Scale to 1000 pcaps/day? Async FastAPI (I/O-bound), Celery (job ID + poll), Qdrant sharding to keep search latency flat.
Re-angle by question: RAG → metadata-prefilter story. HITL/safety → the 0.65-refusal story. Cost/caching → Redis + model-routing story. "An agent" → your full perceive→retrieve→reason→act→escalate loop.

#2LLM Evaluation Framework

FastAPICeleryRedisPostgreSQLStreamlit

They will ask how you QA an AI system. This is the answer.

"As the owner of an LLM-powered product, your worst failure is silent. A model provider ships an update with no changelog — the contract doesn't change, but answers quietly get worse. Nobody notices for days; support tickets are the first signal.

So I built an automated quality gate. Every prompt/response pair runs through four independent evaluators — LLM-as-judge, hallucination detector, RAG faithfulness scorer, consistency checker. Results land in Postgres, failures fire a Slack alert, a live dashboard shows the trend.

The key decision: AND-logic, not a weighted average. A response 95% faithful but hallucinating one claim still fails — an average would hide exactly that failure. It's a quality gate, not a quality summary.

The part I love: the two RAG scores diagnose the failure. High faithfulness + wrong answer means the retriever fed bad documents. Low faithfulness means the model ignored the retriever. Different failure, different fix.

In a simulated provider update, it caught a 14-point faithfulness drop in under four minutes — versus two to three days of manual review."

Four evaluators run independently; the AND gate fails the case if any single one misses threshold — a weighted average would hide exactly the failure that hurts a user.

Go-deep answers

  • Faithfulness vs hallucination? Hallucination = claims unverifiable against a reference (factual grounding). Faithfulness = did it stay inside retrieved context. Faithfulness 1.0 ≠ correct — means the model only used retrieved content.
  • Why Celery over FastAPI BackgroundTasks? Background tasks are in-process — a restart mid-eval (a deploy) silently loses the task, and a PR that should've been blocked passes. Celery persists in Redis and retries.
  • Why async stack? API stays responsive while evals run 5–30s in the background.
  • Cost at scale? Local-model fallbacks (80MB sentence-transformer instead of an API call), evaluators selectable per test case.
  • Extensibility? Registry/factory pattern + a JSON eval_scores column — new evaluator = new class + registered name, no migration.
Re-angle: this is your answer to "how do you evaluate agent quality before release" and "how do you make AI trustworthy." Connect it to their process: "this is exactly how I'd QA a UWM agent before it ships — a golden suite as a blocking gate."

#3Robot Framework Keyword Generator

StreamlitFastAPIClaude HaikuRF Parser

Headline answer to "when should an agent NOT be autonomous?"

"A test engineer describes a test in plain English and gets a ready-to-use Robot Framework keyword in under two seconds instead of an hour of boilerplate. But an LLM will happily generate syntactically beautiful, semantically wrong test code — and a wrong test that passes is worse than no test.

So I split the job by what each part is good at. The LLM does only entity extraction — understanding that 'mobile identity' means IMSI — and deterministic templates generate the structure, syntactically correct by construction. Robot Framework's own parser validates the output before it's ever returned. And a human reviews every keyword before it enters the suite — human-in-the-loop by design.

The fallback chain — LLM to regex to defaults — means it never crashes. Zero to 50-plus generations a week, on word of mouth."

Go-deep answers

  • Why templates, not full LLM generation? Reliability (LLMs botch RF's indentation-sensitive syntax) + cost/latency at 50+/week.
  • Why Haiku specifically? Entity extraction is narrow — no reasoning needed. 3–5× cheaper, ~500ms.
  • Bad JSON from the LLM? Automatic regex fallback — tester sees no error. LLM path is an enhancement, not a dependency.
  • Why FastAPI + Streamlit split? Testable endpoints, reusable /generate (CLI, CI, IDE plugin), UI redesign doesn't touch the engine.

#4NFTRACE — Distributed Network Function Tracing

FastAPIRedisSQLiteXML-RPCmultiprocessing

Your agent memory / lifecycle / "right tool, not more tools" story.

"Our CI/CD pipelines for validating 5G network functions were slow and fragile, and test scenarios were tangled with validation logic — changing one risked breaking the other. Parallel test runs stepped on each other's state.

So I architected NFTRACE — a distributed platform separating what you test from how you validate it, exposed as four REST calls: start, stop, merge, validate.

Two decisions made it work. Lifecycle endpoints turned scenarios into composable building blocks. And the hard part — parallel runs without corrupting each other's state — I gave each run a UUID-keyed session, and split persistence by temperature: Redis for hot active-session state with TTL cleanup if a worker crashes, SQLite for cold completed history that survives restarts.

Cut pipeline validation time in half. And the CD integration caught a whole class of bugs invisible before: tests that passed while the network behavior underneath was actually wrong."

Redis = working memory (hot, TTL). SQLite = episodic memory (cold, persistent). Right store per access pattern — same idea as Tech Deep-Dives §7.

Go-deep answers

  • Redis + SQLite, why both? Redis: sub-ms, concurrent-safe, TTL auto-expires orphaned sessions. SQLite: persistent history, read infrequently. SQLite for hot state under 20 workers bottlenecks on write serialization.
  • Session isolation? UUID Redis key session:{uuid4()}; every op requires the session ID.
  • Why multiprocessing not asyncio? N nodes must start simultaneously; xmlrpc.client is synchronous. Pool used the stdlib without an async dependency.
  • Why SQLite not Postgres? Single-server test lab, sequential writes, low concurrent reads. "Documented exactly when to migrate: multi-server, failover, or hundreds of concurrent writes."

#5Bias & Fairness Auditor

FastAPIClaudeVADERSciPyStreamlit

Use when healthcare/insurance/fintech or HIPAA comes up.

"As a compliance officer, when the EU AI Act requires you to prove your AI isn't biased, you can't hand regulators a shrug — you need documented, reproducible evidence. Manual spot-checking isn't statistically credible.

So I built an auditor that does counterfactual fairness testing — takes a prompt, changes only the demographic signal, runs each variant many times, measures whether outputs differ. Any difference can only be explained by the demographic.

The decision that matters: the LLM-as-judge never sees the demographics — stripped before it judges. Otherwise you'd import the judge's own training-time bias into the audit. Two-layer scoring, ANOVA plus Cohen's d for statistical and practical significance. Output is EU AI Act Article 13-compliant documentation, generated automatically.

That's the same discipline healthcare and insurance need — audit trails, blind evaluation, data minimization. HIPAA changes what data is sensitive, not how you engineer for it."

Go-deep answers

  • Why multiple runs per variant? LLMs are non-deterministic — one run can't distinguish bias from noise.
  • Cohen's d on top of p-value? P-value = is it real; Cohen's d = does it matter.
  • Blind judge? Strip demographics so the judge rates content, not the person.
  • Async + semaphore? 500 calls per audit — semaphore caps in-flight at 10, avoids instant rate-limiting.
