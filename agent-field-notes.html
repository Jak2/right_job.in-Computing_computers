§0 THE ASK

What they actually said

Requirements

  • Three small agents, each gathering/processing info via LLM or API calls
  • Independent or collaborative — decided per task, not fixed in advance
  • A database persists outputs and execution logs
  • A dashboard displays the result
  • Tasks are announced live, on the day — nothing to pre-build for the specific task

The real constraint

Not "can you build an agent" — "can you justify each decision while building it, under a clock." No blind paste; every choice gets explained out loud. Everything below is written for that.

§1 NAIVE APPROACH

"An agent is just a chatbot that's smart enough to figure it out"

This is where almost everyone starts. One prompt in, one LLM response out. Call it "the agent," ship it.

naive shape
Task
→
LLM call
→
Answer
Where it breaks

One call has no memory of its own steps, can't decide to go fetch something it's missing, and can't act on the world (call an API, write a file, query a DB) mid-thought. The moment the task needs "look something up, THEN reason about it, THEN maybe look up something else" — a single call can't do that. It's a function, not an agent.

§2 COUNTER-INTUITIVE FIX

An agent is a loop, not a smarter model

The model doesn't change. What changes is that you wrap it in a cycle: observe the current state → decide what to do next → take one action (call a tool/API, or answer) → observe the result → repeat until a stop condition fires. This is the minimal shape behind almost every agent pattern you'll hear named (ReAct, plan-and-execute, etc) — they're variations on this loop, not something categorically different from it.

minimal agent loop
Observe state
→
Decide next action
→
Act (tool / API / answer)
↺
loops back to Observe until stop condition

Setting a stop condition

Not optional — it's the difference between an agent and a runaway bill. Three ways to set one:

  1. Fixed max-iterations count
  2. A "task complete" signal the model itself emits
  3. A budget (time/tokens/cost) checked every cycle
§3 BOUNDARY

Where the loop itself becomes the wrong answer

The full observe-decide-act loop is built for tasks whose length you can't predict up front — open-ended research, multi-step debugging, anything where the number of steps depends on what the agent finds along the way.

This is where your interview task probably lives, and probably doesn't need it

You're told to build three small agents, not one deep-reasoning loop. That phrasing is a strong signal: each agent is likely a single-shot worker — one well-structured LLM/API call per agent, with clear input and output — not each agent running its own internal loop. Building full ReAct loops into all three, live, under an hour, burns your clock on machinery the task didn't ask for. Reach for a loop inside one agent only if that specific agent's task is genuinely open-ended (e.g. "keep searching until you find X").

§4 MAPPING TO THE INTERVIEW

Three single-shot agents + a shared blackboard

Agents don't need to call each other directly. Every input/output gets written to one shared table — the blackboard — which both the DB-persistence requirement and the dashboard read from directly.

Components

  • Agent — one run(input) → output unit wrapping a single LLM/API call
  • Orchestrator — decides call order, and whether agent B's input includes agent A's output
  • Blackboard — the shared table every agent reads/writes; doubles as log and dashboard source

Wiring — independent vs. collaborative

The orchestrator, not the agents, decides this per task: chain outputs into inputs for collaborative work, or call agents in parallel with the same task input when they're independent.

§6 THE LEANER CUT

Second-guessing my own design

Fair question to ask of any design, including my own: is §4/§5 actually the simplest thing that satisfies the ask, or is it just good engineering habit showing up where it isn't earned? Re-running it against the real constraint — one hour, task unknown until the day, narrated live — it's the latter. Three things in the original plan are ceremony, not requirement.

pieceimmediate costsecond-order costverdict
BaseAgent class + 3 subclasses typing class, self, inheritance live if asked "why OOP here," you're defending a choice that buys nothing — no swapping/testing/extending happens in a one-off demo drop it. plain functions do the same job with one fewer concept to defend
separate schema.sql file one more file to create and open live file-juggling under a clock for zero payoff — CREATE TABLE IF NOT EXISTS is one inline line drop it
server-style dashboard (Streamlit/Flask) pip install, start a server, bind a port, open browser — live the actual risk: network/pip failure or port collision burns your clock on infrastructure, not on the thing being scored too much surface for a 1-hour unknown-task demo
Leaner shape — same requirements, fewer failure points

Three plain functions (agent_1, agent_2, agent_3), each one LLM call + one log_run() call. One shared call_gemini() helper. One shared log_run() helper writing to SQLite inline (stdlib, no schema file). "Dashboard" = a script that queries the DB and writes a static HTML table, opened via file:// — zero server, zero port, zero deps beyond google-generativeai. Every remaining piece maps 1:1 to a literal line in the problem statement.

§8 MINIMAL AGENT — THE ACTUAL SHAPE

What you're typing, stripped to essentials

Base pattern (§4/§5 shape — see §6 for the leaner alternative)

this

class BaseAgent:
    name: str
    system_prompt: str

    def run(self, input: dict) -> dict:
        output = call_llm(
            self.system_prompt, input
        )
        log_to_db(self.name, input, output)
        return output

Three subclasses of this, one orchestrator.py that calls them in whatever order the live task needs, one schema.sql for the runs table, one small dashboard script that queries it. That's the whole system. Nothing here is framework-shaped — every piece is something you can point at and explain.

The real Gemini call (revised from Claude — see §7)

call_gemini — actual Gemini API shape
import google.generativeai as genai

genai.configure(api_key=os.environ.get("GEMINI_API_KEY"))

def call_gemini(system_prompt: str, input: dict) -> str:
    model = genai.GenerativeModel(
        model_name="gemini-1.5-flash",
        system_instruction=system_prompt,
    )
    res = model.generate_content(str(input))
    return res.text
Verify before the call

genai.GenerativeModel(...).generate_content(...) and the res.text shape are the standard google-generativeai SDK surface — confirm against whatever version lands in your pip install google-generativeai on interview day, and check ai.google.dev for the current free-tier model name/limits since gemini-1.5-flash may have been superseded.

§9 WHEN THE AI GETS IT WRONG, LIVE

The actual test isn't the happy path

He said it outright: not communication skill, not memorized syntax — problem-solving, watched in real time. The moment the AI-generated code does something wrong is the moment that's actually being scored. The happy path proves you can prompt; the recovery proves you can think.

Before you run anything

  • Say the expected output out loud before executing — "this should return a JSON object with a summary field." A stated expectation makes any deviation immediately visible and narratable, instead of a silent surprise.
  • Wrap every LLM call's output handling in something that fails loud, not silent — an unhandled exception you can read beats a swallowed error you have to go hunting for.

Failure classes, by what they actually look like

what you seelikely causerecovery move
Response isn't valid JSON / missing a field you expected Prompt didn't constrain the output shape tightly enough Re-prompt with an explicit schema in the system prompt ("respond with JSON: {field: type}"); narrate that you're tightening the contract
Model references a method, API, or fact that doesn't exist Hallucination — models do this confidently, not just when uncertain Say it plainly: "that's not verified, I wouldn't trust this without checking the real API." Don't defend a hallucinated detail because the AI produced it
Script throws on import or syntax Version mismatch, or the AI assumed a library/method that isn't installed Read the actual traceback line out loud, isolate which call failed, fix that one line — don't regenerate the whole file blind
Correct-looking output, but wrong for the task Prompt under-specified the actual goal Narrate the gap between what you asked and what you meant, then tighten the prompt — this is the most common one and the easiest to explain well

The rule that matters most

Never silently retry. Every retry gets a spoken reason: what was wrong, what you changed, why you expect this attempt to work. A silent retry that happens to succeed looks like luck. A narrated retry looks like debugging skill — which is the actual thing being evaluated.

Rehearsal tip

Deliberately break something once during practice — feed an agent a malformed input, or ask the LLM a question it can't answer well — so the recovery move is muscle memory, not improvised for the first time under a clock.

§10 QUESTIONS HE MIGHT ASK

Anticipated, from his side of the table — with answers

Grounded in what he emphasized directly: problem-solving process, on-screen reasoning, no blind copy-paste. Written as if answering him directly, out loud — say these in your own words, not read verbatim.

Architecture & design decisions

"Why three separate agents instead of one doing everything?"

I split it into three because a single LLM call only gets one shot — ask, get an answer, done. It can't look something up, reason about it, and then decide to go look up more. The moment I need that kind of staged thinking, I need separate steps. So each agent owns one narrow job, with a clear input and output I can point at.

"Why didn't you use a framework — LangChain, CrewAI, AutoGPT?"

I didn't reach for a framework because then every retry, every parsing decision, every routing choice is happening inside code I didn't write. If you ask me why something did what it did, I want to answer from my own code, not from library internals I'd have to go look up live. For three small agents in an hour, hand-rolling it is actually faster for me to reason about.

"Why plain functions and not classes?"

I went with plain functions because classes earn their cost when you're swapping implementations or testing over time. This is a one-shot build — a function that takes an input and returns an output does the same job, and it's one less thing I have to justify while I'm also explaining the agent logic itself.

"How do you decide if two agents should be independent or collaborative?"

That's not really the agents' decision, it's mine, in the orchestrator. If agent B actually needs agent A's result to do its job, I chain them — A's output becomes part of B's input. If they don't depend on each other, I just call them in parallel with the same task. I look at the task first, then decide the wiring.

"Walk me through what happens, end to end, for the task you just gave me."

Sure — the task comes in, the orchestrator decides how the three agents should be wired for it. Each agent makes one call to Gemini, and right after that I log the input and output into a SQLite table. Once all three have run, the dashboard just reads that table and shows the trace. Same flow no matter what the specific task ends up being.

"What would you cut first if you only had 20 minutes instead of 60?"

The dashboard polish. Even just printing the database rows to the terminal proves the persistence works and shows you a result. What I won't cut is the agent logic and the database writes — those are the actual requirements. Presentation is the one thing I'd trade for time.

"Why haven't you implemented a control loop — shouldn't an agent be able to trigger itself again if needed?"

That's a deliberate call, not an oversight. A loop — observe, decide, act, repeat — earns its cost when the number of steps isn't known upfront: open-ended search, retrying until an output passes validation, debugging until tests pass. My three agents are single-shot. Each one takes a defined input, makes one call, produces a defined output, and hands it to the next stage. There's no branching decision inside any of them that requires re-evaluating state and choosing an action again — so a loop here would just be machinery with nothing to do, extra code I'd have to justify without it buying any correctness or robustness. If one of your three tasks turns out to be genuinely open-ended — keep searching until a specific fact turns up, or validate an output and retry until it's well-formed — that's exactly the signal to add a loop, but only to that one agent, not all three. I'd rather add that complexity the moment it's actually earned than pre-build it on a guess.

Data & persistence

"Why SQLite and not Postgres, Mongo, or just a file?"

SQLite, because it's zero setup — it's one file. If you want to see what's actually stored, I can open a terminal and query it in one line, right in front of you. A networked database would just be one more thing that could fail to connect, for no benefit at this scale.

"What exactly gets logged — inputs, outputs, timestamps, errors?"

For every single agent call I log the agent's name, the full input it received, the full output it produced, and a timestamp. And that's true even when a call fails — I don't want failures disappearing silently, I want them sitting in the same table as everything else.

"If this ran 1,000 times a day in production, what would you change first?"

Honestly, a few things — SQLite doesn't handle concurrent writes well at that volume, so I'd move to something that does. I'd add retry and backoff around the LLM calls instead of handling failures manually. And I'd separate the execution log from anything user-facing. None of that's needed for a one-hour demo, but I want to be clear I know where that line is.

Failure & edge cases

"What happens if agent 2 depends on agent 1's output, and agent 1 returns garbage?"

Then agent 2 is going to work off garbage too — but because I'm logging both, that's visible immediately in the dashboard, I'm not hunting for it. My fix in that moment is to go back and tighten agent 1's prompt, not to try patching it downstream in agent 2.

"How do you know the LLM's output is actually correct, not just well-formatted?"

Honestly, I don't, by default. Well-formatted and correct are two different claims. If it's something checkable — a fact, a number — I'd verify it against a real source. If it's more open-ended synthesis, I'd just say plainly that this hasn't been independently verified, rather than pretend it has.

"What if the API call times out or hits a rate limit mid-demo?"

If that happens, I'm not going to pretend it didn't. I'll say out loud what just happened — that's a timeout or a rate limit — and then retry with a reason, not just silently hit run again and hope.

"Two agents produce conflicting results — how do you resolve that?"

I wouldn't quietly pick one. I'd log both outputs, point out that they disagree, and either re-run the next agent with both flagged, or treat the disagreement itself as a sign that one of the prompts needs tightening.

Live AI-tool use (the one he's most likely probing hardest)

"Explain this specific line the AI just wrote — what does it do, and why is it there?"

I'd walk through it line by line in my own words — not repeat back whatever comment the AI put there. If I get to a line and can't explain what it does, that's my signal to stop and actually read it before I keep going.

"How do you know the AI's own explanation of its code is accurate?"

I don't just take its word for it. I check its explanation against what the code actually does when I run it — same as I wouldn't trust a comment in a teammate's code without reading the function underneath it.

"That didn't work — what's your next move?"

First thing I do is say out loud what kind of failure this is — bad output format, a made-up method that doesn't exist, an actual traceback, or a prompt that was too vague. Then I make one specific fix for that. I don't just regenerate the whole thing and hope it's different this time.

"Would you trust this output if it were going in front of a customer?"

Not as-is, no. What I've built here is optimized to show my reasoning clearly in an hour, not to be production-ready. Those are different bars, and I'd rather say that directly than oversell it.

Meta / reflection

"What's the biggest risk in this design, and how would you mitigate it?"

The biggest risk is one call in the middle returning something malformed and breaking the chain for whatever comes after it. I mitigate that by keeping each agent's expected output format tight in the prompt, and by logging every call so a bad one is visible the moment it happens, instead of quietly poisoning the next step.

"What would you have done differently with more time?"

With more time I'd add actual schema validation on the outputs instead of just trusting the text back, and a real retry-with-backoff layer instead of me manually re-prompting. I skipped both on purpose for today, not because I didn't think of them.

"Why this architecture over the alternatives you considered?"

I actually built a more class-based version too, with its own schema file and a Streamlit dashboard, and I can walk you through why I moved away from it for this — it adds a server and more moving parts, which is more that can fail live, for no benefit at this scope. I went with the leaner version because it matches the actual constraint: one hour, and the task decided on the spot.

§11 BEFORE THE CALL

Checklist

  • Base agent class + one working single-shot call tested against a throwaway prompt, before the interview
  • SQLite schema created and one row written + read back, end to end
  • Dashboard renders from the DB, not from in-memory state — proves persistence actually works
  • A stop-condition habit: say out loud, for whatever loop you write, "this stops when ___"
  • Rehearsed one practice task through the full pipeline so the wiring is muscle memory, not improvisation
